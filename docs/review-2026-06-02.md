## 1. 화면 리뷰 (UX)

### 1.1 대시보드 홈 `/` — ★★★★☆
**좋은 점**
- 첫 진입에서 통계 카드 + 최근 활동 + 빠른 시작 — SaaS 대시보드 문법을 잘 따름
- 빈 상태 온보딩("커리어 데이터부터 등록")이 친절

**개선**
- 통계가 클라이언트 3-API 병렬 호출 집계 → 데이터 많아지면 느려짐. 배포 시 집계 엔드포인트 필요
- 합격률 "—" 표시는 직관적이나, 0건일 때 "아직 결과 없음" 같은 설명 툴팁이 있으면 더 좋음

### 1.2 워크플로우 빌더 `/generate` — ★★★★☆ (핵심 화면)
**좋은 점**
- JD분석 → 항목별 병렬 파이프라인 fan-out 시각화는 이 프로젝트의 **시그니처**. 멀티에이전트를 "말이 아니라 화면으로" 증명
- 노드 인라인 모델 설정 + 실시간 상태 체인(파랑→초록) + N회차 뱃지 — 동작이 보이는 UI
- 사이드바(입력) + 캔버스(시각화) + 결과 패널(하단) 3분할이 합리적

**개선 (우선순위 순)**
1. **모바일/좁은 화면 대응 없음** — 풀스크린 캔버스는 데스크톱 전용. 반응형 분기 필요
2. **항목 0개일 때 캔버스가 추상 5노드** — 사용자가 "왜 항목이 안 보이지?" 혼란 가능. 안내 문구 강화
3. **생성 취소 버튼 부재** — 60초+ 걸리는데 중단 불가 (SSE abort)
4. **톤/페르소나는 전역 공통** — 모델은 항목별인데 톤은 아니라 일관성 깨짐

### 1.3 모델 & API `/models` — ★★★★☆ (방금 개선됨)
**좋은 점**
- GPU 배너 + VRAM fit 경고 — 하드웨어 인식 UX는 로컬 LLM 도구의 차별점
- 미설치 모델 우선 정렬, API 키 명시적 저장 버튼 (이번 개선)

**개선**
- 추천 모델 다운로드 중 진행률이 카드 안 작은 바 → 큰 모델(5GB+) 다운로드 시 체감 느림. ETA 표시 고려
- API 키 입력란이 password type이라 붙여넣기 후 확인 불가 — 눈 아이콘(show/hide) 토글 권장

### 1.4 공통 UX 부채
- **`alert()` 9곳** — 브라우저 기본 alert은 조악함. Toast 통일 필요 (ROADMAP M6)
- **로딩 상태 제각각** — 일부 "불러오는 중...", 일부 스피너. 스켈레톤 통일
- **한국어 폰트 미적용** — Geist(영문 폰트)라 한글 렌더링이 최적이 아님. Pretendard 권장
- **다크 모드 없음**

## 2. 코드 리뷰

### 2.1 잘한 점 (유지할 것)
- **타입 안전성**: 프론트 `any` 0건, 백엔드 Pydantic 전면 적용
- **결정적 로직 테스트**: char_counter/text_cleaner/tech_extractor + smoke/RAG 통합 = 46건
- **레이어 분리**: `api/v1` (라우팅) → `services` (비즈니스) → `agents`/`rag` (도메인). 책임 명확
- **LLM Factory 레지스트리** (ADR-011): 프로바이더 추가가 한 곳 등록으로 끝남
- **graceful degradation**: GPU 조회 실패 시 경고 비활성화 등 방어적 설계
- **ADR 문화**: 25건의 의사결정 기록 — 1인 프로젝트에서 드문 수준

### 2.2 🔴 High — 보안 (M5 잔여, Rule #2 위반 상태)
1. **`crypto.py`(Fernet)가 아무 데서도 import 안 됨**
   - API 키 DB 암호화 유틸을 만들어두고 **연결 안 함**. 현재 키는 프론트 localStorage 평문 저장.
   - CLAUDE.md Rule #2("API 키 AES-256 암호화 후 DB 저장")가 **선언만 되고 미구현**.
   - → `UserLLMConfig` 테이블 + 라우터 연결 필요. (단, 현재 단일 사용자라 실위험은 낮음)
2. **`ENCRYPTION_KEY` 검증 없음**
   - `config.py`에 default `"change-me-in-production"` 그대로. startup 시 기본값이면 에러를 띄워야 함.
3. **CORS가 `localhost:3000` 하드코딩** — 배포 시 환경변수 분기 필요 (현재는 개발용으로 OK)

### 2.3 🟡 Medium — 아키텍처 / 일관성
1. **선언한 스택 vs 실제 구현 간극** (포트폴리오 관점 중요)
   - `requirements.md`/CLAUDE.md: "**React Query**(서버 상태) + **Zustand**(클라이언트 상태)"
   - 실제: 전부 `useState` + `useEffect` + 수동 `fetch` (20곳). React Query/Zustand **미사용**.
   - → 면접에서 "왜 선언했는데 안 썼나?" 질문 위험. **둘 중 하나**: 실제 도입하거나, 문서에서 빼고 "의도적으로 경량 유지"로 정직하게 기술.
2. **`generate/page.tsx` 774줄 / useState 25개** — 거대 컴포넌트
   - 상태 관리 복잡도 높음. 커스텀 훅(`useEssayGeneration`, `useAgentConfig`)으로 분리 권장.
   - React Query 도입 시 서버 상태(jobs/library/projects)가 자연히 빠지며 슬림해짐.
3. **`essays`는 서비스 레이어 없이 라우터에서 orchestrator 직접 호출**
   - jobs/library/projects는 `services/` 거치는데 essays만 예외. 일관성 위해 `services/essay.py` 고려 (선택).

### 2.4 🟡 Medium — 에러 핸들링
1. **빈 `except Exception:` 7곳** — 일부는 의도된 graceful(gpu.py, rag_retriever)이나, `url_fetcher.py`·`file.py`는 원인을 삼킬 위험. 최소한 로깅 추가.
2. **에러 응답 형식 불일치** — `HTTPException` vs `ValueError` vs 생성 SSE error 이벤트가 혼재. 표준 에러 스키마 정의 권장.
3. **구조화 로깅 부재** — `structlog` 미도입. 디버깅·운영 시 추적 어려움 (ROADMAP에 있음).

### 2.5 🟢 Low — 정리
- E501(줄길이) 23건 — 한국어 주석/docstring. ruff `line-length`를 한국어 고려해 조정하거나 개별 정리
- 글자수 초과 근본 미해결 (500자→689자) — 인라인 편집 워크어라운드는 있으나 LLM 한계
- 클라우드 모델 목록 하드코딩 — API 조회는 키 필요라 트레이드오프

## 4. 결론

**현재 상태로도 "멀티에이전트 오케스트레이션을 시각화한 풀스택 AI 도구"로서 포트폴리오 가치가 충분하다.** 특히 워크플로우 빌더와 ADR 문서화는 강점.

포트폴리오 완성도를 한 단계 올리는 **가장 비용 대비 효과 큰 3가지**:
1. **보안 마무리** (ENCRYPTION_KEY 검증 + 키 암호화) — "Rule #2를 선언만 하고 안 지킴"은 리뷰어가 가장 먼저 잡는 지점
2. **스택 정합성** (React Query/Zustand 도입 or 문서 정정) — 선언과 구현 일치
3. **`generate` 컴포넌트 분할** — 774줄은 유지보수성 면접 질문 1순위

이 셋만 처리하면 코드 품질 평가가 ★★★★☆ → ★★★★★ 수준으로 올라간다.
