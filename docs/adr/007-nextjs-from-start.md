# ADR-007: 처음부터 Next.js 채택 (Jinja2 단계 생략)

- **상태**: 채택 (Accepted)
- **날짜**: 2026-05-22
- **결정자**: 개발자
- **관련 요구사항**: 기술 스택 (프론트엔드)

---

## 컨텍스트

FastAPI 프로젝트에서 UI를 빠르게 만드는 방법으로 두 가지를 검토했다:
1. **Jinja2 템플릿** (FastAPI 내장, 서버 사이드 렌더링, 빠른 시작)
2. **Next.js + TypeScript** (분리된 프론트엔드, React 생태계)

초기 v0.1 계획은 Jinja2로 시작 → Next.js 마이그레이션이었으나,
v0.2에서 처음부터 Next.js로 결정을 변경했다.

---

## 결정

**Jinja2 단계를 생략하고 처음부터 Next.js 14 (App Router) + TypeScript를 사용한다.**

---

## 이유

| 비교 항목 | Jinja2 먼저 | Next.js 직행 |
|---------|-----------|-------------|
| 초기 속도 | 빠름 | 약간 느림 |
| 마이그레이션 비용 | 높음 (HTML→React 전환) | 없음 |
| 채용 시장 가치 | 낮음 | 높음 (React 필수) |
| Claude Code 친화성 | 보통 | 높음 (React 최적화) |
| Phase 2 기능 확장 | React Flow 추가 어려움 | 자연스러운 확장 |
| 포트폴리오 임팩트 | 낮음 | 높음 |

핵심: Jinja2→Next.js 마이그레이션 비용이 처음부터 Next.js로 시작하는 것보다 크다.
      결국 Next.js를 써야 한다면, 처음부터 하는 게 낫다.

---

## 대안 (검토 후 기각)

### 대안 1: Jinja2 → Next.js 순차 마이그레이션 (v0.1 계획)
- ❌ 기각: 마이그레이션 공수 = 두 번 개발하는 것과 같음

### 대안 2: React (CRA / Vite)
- ❌ 기각: Next.js가 라우팅, SSR, 최적화 내장

### 대안 3: Streamlit (빠른 데모)
- ❌ 기각: 커스터마이징 한계, 포트폴리오 가치 낮음

---

## 결과

### 긍정적
- ✅ 마이그레이션 비용 없음
- ✅ 채용 시장에서 React/Next.js 경험 증명
- ✅ Phase 2 React Flow (워크플로우 시각화) 자연스럽게 추가 가능
- ✅ shadcn/ui로 디자인 부담 최소화
- ✅ Claude Code가 React/Next.js에 최적화되어 AI 도움 극대화

### 부정적
- ⚠️ 초반 TypeScript, Next.js App Router 학습 필요
- ⚠️ Jinja2 대비 초기 설정 시간 더 걸림
