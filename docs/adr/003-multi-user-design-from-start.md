# ADR-003: 처음부터 멀티유저 설계 강제

- **상태**: 채택 (Accepted)
- **날짜**: 2026-05-22
- **결정자**: 개발자

---

## 컨텍스트

Phase 1은 개인 도구(단일 사용자)이지만, Phase 3에서 멀티유저 SaaS로 확장할 계획이다.
단일 사용자용으로 시작했다가 나중에 멀티유저로 마이그레이션하는 것과,
처음부터 멀티유저 설계를 적용하는 것 중 선택이 필요했다.

---

## 결정

**단일 사용자여도 모든 DB 테이블과 쿼리에 `user_id`를 포함한다.**

```python
# 모든 모델에 user_id 강제
class CareerDocument(Base):
    user_id: Mapped[str] = mapped_column(index=True)  # 필수

# 모든 쿼리에 필터 강제
documents = session.query(CareerDocument).filter_by(user_id=current_user_id).all()
```

---

## 이유

Phase 3 배포 시 단일→멀티유저 마이그레이션 비용이 매우 높다:

| 항목 | 단일→멀티 마이그레이션 비용 |
|------|---------------------------|
| DB 스키마 변경 | 모든 테이블에 `user_id` 컬럼 추가 + 마이그레이션 |
| 기존 데이터 처리 | 기존 레코드에 user_id 값 어떻게 채울지 결정 |
| API 수정 | 모든 엔드포인트에 인증/user_id 필터 추가 |
| 버그 위험 | 누락된 필터 → 데이터 유출 |

처음부터 적용하면 추가 복잡도는 최소, 장기 이득은 크다.

---

## 대안 (검토 후 기각)

### 대안 1: Phase 3에서 마이그레이션
- ❌ 기각: 마이그레이션 공수 크고 데이터 유출 버그 위험

### 대안 2: user_id 없이 row-level security만
- ❌ 기각: PostgreSQL RLS는 복잡도 높고 테스트 어려움

---

## 결과

### 긍정적
- ✅ Phase 3 배포 시 재설계 불필요
- ✅ 데이터 격리 버그 없음
- ✅ RAG 검색도 처음부터 `user_id` 메타데이터 필터 포함

### 부정적
- ⚠️ 초기 복잡도 약간 증가 (모든 쿼리에 필터 추가)
- ⚠️ Phase 1에서는 하드코딩된 `user_id = "default"` 사용
