# ADR-025: 항목별 독립 에이전트 설정

- **상태**: 채택 (Accepted)
- **날짜**: 2026-06-01
- **결정자**: 개발자
- **관련**: [ADR-008](008-multi-llm-provider.md), [ADR-011](011-llm-factory-registry-pattern.md), [ADR-015](015-langgraph-send-item-subgraph.md), [ADR-024](024-react-flow-workflow-builder.md)

---

## 컨텍스트

ADR-008/011로 에이전트별(jd_analyzer/essay_writer/compressor/evaluator) 모델 선택은 가능했지만,
설정은 **전역**이었다. 즉 모든 자소서 항목이 동일한 모델 조합을 사용.

### 실사용 요구

- "자기소개는 품질이 중요하니 Claude로, 지원동기는 빠르게 로컬 exaone으로" 처럼
  **항목마다 다른 모델**을 쓰고 싶음.
- 비용/품질 트레이드오프를 항목 단위로 조절 (긴 핵심 항목은 큰 모델, 짧은 항목은 경량 모델).
- ADR-024 워크플로우 빌더에서 각 항목이 독립 행으로 보이니, 행마다 설정이 자연스러움.

---

## 결정

**`EssayItem`에 선택적 `agent_config`를 추가. 항목별 설정이 있으면 전역 설정을 덮어쓴다.**

### 데이터 흐름

```
EssayItemRequest.agent_config (선택)        # API 입력
  → EssayItem.agent_config                  # 항목별 오버라이드 보관
  → _fan_out: item.agent_config or 전역     # 항목별 우선, 없으면 전역
  → ItemState.agent_config                  # 서브그래프가 사용
```

### 병합 규칙

- 항목 config는 **전역 config 위에 부분 병합**: `{**global, **item}`
  → 항목이 `essay_writer`만 지정하면 나머지(jd_analyzer/compressor/evaluator)는 전역 유지.
- JD 분석은 항목 fan-out **이전** 단계라 전역 설정만 적용 (항목별 의미 없음).

### 프론트 (ADR-024 워크플로우 빌더)

- 노드별 `<select>`로 항목 config 편집 → 전역과 다르면 `★ 항목 전용 설정` 뱃지.
- 항목 오버라이드가 없으면 전역값 표시 (편집 시 그때 오버라이드 생성).

---

## 이유

- **부분 병합**: 항목마다 4개 에이전트를 다 지정하는 건 번거로움. 바꾸고 싶은 것만 덮어쓰기.
- **전역 fallback 유지**: 기존 동작(전역 설정) 100% 하위호환. `agent_config` 미지정 항목은 그대로.
- **JD 분석 제외**: fan-out 이전 공통 단계라 항목별 분기가 구조적으로 불가능.

---

## 트레이드오프

| 항목 | 결정 |
|------|------|
| 동시 다중 모델 로딩 | 항목마다 다른 Ollama 모델 → VRAM에 여러 모델 스왑 → 느려질 수 있음 (로컬 한정) |
| 설정 복잡도 | 전역 + 항목별 2단계 → 노드 뱃지로 "전용 설정" 여부 명시해 혼란 완화 |
| API 페이로드 | 항목마다 config 중복 가능 → 부분 병합으로 최소 필드만 전송 |

---

## 명시적으로 하지 않는 것

- ❌ JD 분석 항목별 설정 (fan-out 이전 단계)
- ❌ 항목별 설정 영구 프리셋 저장 (현재는 생성 세션 단위)

## 결과

### 긍정적
- ✅ 자기소개=exaone3.5:7.8b / 지원동기=qwen2.5:7b E2E 검증 — 각각 다른 모델로 정상 처리
- ✅ 전역 설정만 쓰던 기존 흐름 하위호환
- ✅ ADR-024 노드 UI와 1:1 대응 (행마다 모델 선택)

### 부정적
- ⚠️ 로컬에서 항목마다 다른 큰 모델 지정 시 VRAM 스왑으로 지연
- ⚠️ 설정 상태가 전역/항목 2곳에 분산 → 프론트 상태 관리 복잡도 증가
