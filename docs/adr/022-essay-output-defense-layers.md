# ADR-022: 자소서 출력 다층 방어 (할루시네이션 + 폭주 + 다국어 혼용)

- **상태**: 채택 (Accepted)
- **날짜**: 2026-05-27
- **결정자**: 개발자
- **관련**: [ADR-001](001-char-count-validation.md), [ADR-015](015-langgraph-send-item-subgraph.md), [ADR-021](021-tech-stack-auto-extraction.md)

---

## 컨텍스트

M5 실사용 검증에서 자소서 품질 관련 심각한 사례 다수 발견.

### 발견된 실패 모드

**1. 할루시네이션 — 공고 기술을 본인 경험으로 옮김**
- 공고: "Kafka 우대" → 본인은 Kafka 안 썼는데 자소서에 "Kafka 기반 시스템 개발" 작성
- 공고 분석 결과가 essay_writer 프롬프트에 포함되니까 모델이 본인 경험과 혼동

**2. 마크다운/메타 오염**
- `**볼드**`, `- 불릿`, `### 헤더`, `[참고 경험 N]` 출처 레이블 본문 누출
- compressor가 essay_writer가 제거한 마크다운을 다시 도입 (연쇄 버그, v0.7.1에서 일부 수정)
- `<think>...</think>` reasoning 모델 사고 과정 노출 (deepseek-r1)
- `[현재 자소서]` 같은 prompt 라벨이 본문에 echo

**3. 다국어 혼용 — 한국어 자소서에 한자/베트남어/키릴/영문 헤더**
- qwen2.5/deepseek-r1 같은 다국어 모델이 한국어 모드 이탈
- 예: "다른 hiểu士들과 협업하며 빠르게 адаптиров하고..."
- 영문 헤더 "Key Contributions:", "Server Architecture Design & Optimization:"

**4. 토큰 루프 폭주**
- 작은 모델이 동일 패턴 반복: `"8-8-8-8-8-8-8-8-..."`
- max_tokens 다 소진할 때까지 무한 출력

이 모든 실패가 한 번에 발생하면 1시간 넘는 무한 처리 + 결과는 쓰레기. ADR-015 압축 루프 + 본 ADR의 폭주가 결합하면 디버깅 불가.

## 결정

**단일 방어로는 불가능. 3-Layer 방어 + 1개 회복 조치를 동시 적용.**

### Layer 1: Prompt Engineering (예방)

`essay_writer.py` 시스템 프롬프트 강화:

```
[공고 분석]은 회사가 무엇을 원하는지에 대한 참고일 뿐.
[경험 자료]만이 사용자가 실제로 한 경험.

⚠️ 핵심 원칙 (절대 위반 금지):
1. 본인 경험으로 서술할 수 있는 것은 오직 [경험 자료]에 명시된 내용뿐.
2. 공고에 언급된 기술이라도 [경험 자료]에 없으면 본인이 다룬 것처럼 절대 쓰지 말 것.
   ❌ 공고에 "Kafka 우대" + 경험 자료에 Kafka 없음 → "Kafka 사용 경험" 금지
   ✅ 우회: "메시지 큐와 이벤트 기반 처리에 관심" 같은 인접 표현
3. 모든 수치·회사명·기술명·프로젝트명은 [경험 자료]에서만 인용.
```

### Layer 2: 명시적 화이트리스트 주입 (강제 안내)

자연어 지시는 약한 모델이 못 따라감. **사용자가 실제 다룬 기술 목록을 프롬프트에 직접 박는다**:

```
[본인이 실제로 다룬 기술 — 화이트리스트]
이 목록에 있는 기술만 본인 경험으로 서술 가능. 이 목록에 없는 기술은
공고에 언급되더라도 본인이 다룬 것처럼 절대 쓰지 말 것.

Python, FastAPI, PostgreSQL, LangGraph, Docker, ...
```

- `rag_retriever_node`가 사용자의 모든 `CareerDocument.tech_stack` 합집합을 수집해 `ItemState.tech_whitelist`에 저장
- ADR-021의 자동 추출이 화이트리스트의 정확도/완전성을 담보

### Layer 3: 출력 후처리 (회복)

`app/utils/text_cleaner.py` 강화:

| 정리 대상 | 정규식 |
|----------|--------|
| `<think>...</think>` 블록 | `_THINK_BLOCK_RE` |
| `[현재 자소서]` 등 prompt label echo | `_PROMPT_LABEL_RE` |
| `Key Contributions:` 영문 헤더 | `_ENGLISH_HEADER_RE` |
| `**볼드**`, `*이탤릭*` | `_BOLD_RE`, `_ITALIC_RE` |
| `- ` `* ` 불릿 마커 | `_BULLET_RE` |
| `[참고 경험 N]` | `_RAG_LABEL_RE` |
| `**400자**` 글자수 메타 | `_BOLD_META_RE` |
| `"수정 후 글자 수: 400자"` | `_CHAR_META_LINE_RE` |
| `"8-8-8-8-..."` 반복 폭주 | `_REPETITION_RE` → 첫 인스턴스로 압축 |

essay_writer + compressor 모두 `clean_llm_output()` 적용.

### Recovery: 품질 검사 + SSE 경고 (감지)

`detect_output_issue()`가 항목 완료 후 자동 검사:

```python
def detect_output_issue(text: str) -> str | None:
    if _REPETITION_RE.search(text): return "repetition"
    korean_ratio = ...
    if korean_ratio < 0.3: return f"non_korean ({korean_ratio:.0%})"
    foreign_ratio = chinese_cyrillic_vietnamese / total
    if foreign_ratio > 0.05: return f"foreign_script ({foreign_ratio:.0%})"
    return None
```

`orchestrator._process_item`이 결과 검사 후 문제 발견 시 SSE progress에 추가:

```
✅ 자기소개 완료 (514자, 평가 7점) [RAG 5개 참고]
⚠️ 자기소개 품질 경고: foreign_script (14% 한자·키릴 등) — 한국어 특화 모델(exaone3.5:7.8b 등) 사용 권장
```

→ 사용자가 어느 항목이 깨졌는지 즉시 인지, 모델 변경 가이드 제공.

## 이유

### 왜 모델 교체만으로 해결 안 되나

- 사용자가 모델 자유 선택 (멀티 LLM 정책, ADR-008/011) → 다국어/작은 모델 사용 가능성 항상 존재
- 큰 한국어 모델(exaone3.5:7.8b 등)도 가끔 마크다운/영문 헤더 출력
- 비용/속도 트레이드오프 — 사용자가 의도적으로 가벼운 모델 선택할 수 있음

### 왜 단일 방어 안 되나

- **프롬프트만**: 약한 모델은 자연어 지시 무시
- **화이트리스트만**: 모델이 화이트리스트를 못 보거나 우회
- **후처리만**: 패턴이 모든 변형을 못 잡음 + 의미 수준 할루시네이션 못 막음
- **검사만**: 문제 감지해도 이미 본문은 망가짐

→ **다층 방어가 누적되어야** 실패 확률이 곱셈으로 감소.

### 왜 출력 검사는 차단이 아니라 경고인가

자동 재시도/차단 시:
- LLM 호출 추가 비용/시간
- "사용자 입장에서 왜 결과가 안 나오는지" 모호함
- 모델이 영구적으로 망가진 경우 무한 재시도

경고 + 사용자 결정:
- 깨진 결과라도 보여줌 → 사용자가 정도 판단
- "모델 변경 권장" 가이드로 근본 해결 유도
- ADR-015의 압축 루프(최대 3회)와 책임 분리

## 트레이드오프

| 항목 | 결정 |
|------|------|
| 프롬프트 길이 증가 | Layer 1+2로 시스템 프롬프트 +500 토큰 — 큰 모델엔 부담 적음, 작은 모델 컨텍스트엔 비용 |
| 화이트리스트 false negative | 사용자가 안 쓴 기술이 자동 추출되면 (ADR-021 한계) 화이트리스트 오염 — 사용자 검토로 보완 |
| 후처리 over-strip | 정상 본문이 영문 헤더로 오인되면 잘림 — 임계값 조정으로 완화 |
| 검사 false positive | 단어 일부 영문이라도 "non_korean" 경고 — 30% threshold로 일반 자소서엔 영향 적음 |

---

## 명시적으로 하지 않는 것

- ❌ 자동 재시도 (사용자 통제권 우선)
- ❌ 모델별 하드코딩 분기 (예: deepseek-r1만 특수 처리) — 정책은 모델 무관
- ❌ 출력 자동 차단 — 사용자가 결과를 봐야 모델 적절성 판단 가능

## 결과

### 긍정적
- ✅ 마크다운/prompt echo/영문 헤더/반복 패턴 자동 제거 검증
- ✅ Kafka 같은 명백한 할루시네이션 사례에서 화이트리스트가 동작
- ✅ SSE 경고로 사용자가 모델 선택 문제임을 즉시 인지
- ✅ ADR-015 압축 루프와 함께 동작 (압축 시도도 동일 후처리 적용)

### 부정적
- ⚠️ Layer 1+2 합쳐 프롬프트 +500 토큰 → 작은 컨텍스트 모델엔 부담
- ⚠️ 의미 수준 할루시네이션(없는 사실 자연스럽게 지어냄)은 여전히 LLM 한계
- ⚠️ 사용자가 SSE 경고를 무시하면 깨진 결과를 저장할 수 있음

### 권장 사용
- 모든 에이전트에 한국어 특화 모델 통일 (exaone3.5:7.8b 권장)
- 화이트리스트 정확도를 위해 ADR-021 자동 추출 결과 청크 미리보기에서 검토
- SSE 경고 발생 시 모델 변경 후 재생성
