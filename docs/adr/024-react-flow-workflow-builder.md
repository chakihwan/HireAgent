# ADR-024: 자소서 생성 UI를 React Flow 워크플로우 빌더로 전환

- **상태**: 채택 (Accepted)
- **날짜**: 2026-06-01
- **결정자**: 개발자
- **관련**: [ADR-006](006-langgraph-orchestration.md), [ADR-012](012-sse-streaming-response.md), [ADR-015](015-langgraph-send-item-subgraph.md), [ADR-025](025-per-item-agent-config.md)

---

## 컨텍스트

`/generate`는 4-step 위저드(공고 입력 → 항목 선택 → 진행 → 결과)였다. M5 실사용 중 한계가 드러남.

### 문제

1. **멀티에이전트/병렬 처리가 안 보임** — 백엔드는 JD분석 → 항목별 fan-out(RAG→작성→압축→평가)으로
   병렬 동작하는데, UI는 선형 진행 로그만 보여줘서 이 구조가 제품의 핵심 차별점인데도 드러나지 않음.
2. **에이전트 설정이 분리됨** — 모델/프로바이더는 `/settings`에서 전역으로만 설정. 어느 단계에서
   어떤 모델이 도는지 생성 화면에서 알 수 없음.
3. **화면이 좁음** — `max-w-3xl` 중앙 정렬이라 단계별 정보가 세로로 길게 쌓임. Dify·n8n 같은
   노드 그래프 도구의 "한눈에 파이프라인 파악" 경험과 대비됨.
4. 포트폴리오 관점에서 "멀티에이전트 오케스트레이션"을 시각적으로 증명할 화면이 없었음.

---

## 결정

**`/generate`를 풀스크린 워크플로우 빌더로 전면 재설계. 그래프는 React Flow(`@xyflow/react` v12)로 렌더.**

### 레이아웃

```
┌──────────────┬────────────────────────────────────────┐
│ 사이드바(320) │  React Flow 캔버스 (풀스크린)            │
│ - 공고 입력   │   [JD분석] ┬→ [자기소개: RAG→작성→압축→평가]│
│ - 항목 선택   │            └→ [지원동기: RAG→작성→압축→평가]│
│ - 톤/페르소나 │   (드래그·줌·미니맵·컨트롤)               │
│ - 진행 로그   ├────────────────────────────────────────┤
│ - 생성 버튼   │  결과 패널 (생성 완료 시 하단 슬라이드업) │
└──────────────┴────────────────────────────────────────┘
```

### 그래프 두 모드

- **항목 미선택**: 5-노드 추상 파이프라인 (JD분석→RAG→초안→압축→평가) — 설정 편집용
- **항목 선택 후**: JD분석 노드에서 항목 수만큼 **fan-out**, 각 항목이 독립 행(`RAG→작성→압축→평가`)
  → ADR-015의 `Send` API 병렬 구조를 UI에 1:1 대응

### 실시간 상태 (ADR-012 SSE 확장)

- 백엔드 sub-node가 `node_event` SSE 이벤트 emit: `{node, category, phase, detail, iteration}`
- 프론트는 `processedRef`로 미처리 이벤트를 **배치 처리** → 노드/엣지 상태 갱신
  - 노드: 대기(회색) → 실행(파랑 pulse) → 완료(초록) / 오류(빨강)
  - 압축 반복 시 `N회차` 뱃지
- 동시에 추가되는 이벤트(예: rag done + write start)를 `events[length-1]`만 보면 누락 →
  `slice(processedRef.current)`로 전부 순서대로 적용

### 노드 인라인 인터랙션

- 각 노드 카드 안에서 provider·model을 `<select>`로 직접 변경
- React Flow가 노드 영역 클릭을 드래그/팬으로 가로채는 문제 → 인터랙티브 요소에 `nodrag nopan` 클래스

## 이유

### 왜 React Flow인가

- 노드/엣지/줌/팬/미니맵을 직접 구현하는 비용이 큼. React Flow는 사실상 표준(n8n·Dify·Flowise 사용).
- React 19 + Next 16과 호환되는 v12 사용 가능.
- 커스텀 노드(`nodeTypes`)로 우리 도메인(에이전트 설정 카드) 자유 렌더.

### 왜 풀스크린인가

- 항목 N개 × 4단계 = 최대 수십 개 노드. 좁은 컨테이너로는 병렬 구조가 안 보임.
- 사이드바 입력 + 캔버스 시각화 분리가 워크플로우 도구의 검증된 패턴.

### 왜 4-step 위저드를 버렸나

- 단계 전환마다 화면이 통째로 바뀌어 "지금 무슨 일이 일어나는지" 맥락이 끊김.
- 설정(어떤 모델)과 실행(진행 상태)을 같은 화면에서 봐야 디버깅·튜닝이 쉬움.

---

## 트레이드오프

| 항목 | 결정 |
|------|------|
| 번들 크기 | `@xyflow/react` +~150KB — 핵심 화면이라 수용 |
| SSR | React Flow는 클라이언트 전용(`"use client"`). hydration 주의 (localStorage 설정은 `useEffect` 지연 로드) |
| 학습 곡선 | nodeTypes/handle/edge API 학습 필요 — 1회성 비용 |
| 이벤트 동기화 | SSE 이벤트와 그래프 상태를 수동 매핑 → 배치 처리로 누락 방지 (`processedRef`) |
| 모바일 | 풀스크린 캔버스는 데스크톱 우선 — 모바일 반응형은 후순위 |

---

## 명시적으로 하지 않는 것

- ❌ **사용자가 노드를 자유롭게 추가/삭제/연결** — 파이프라인 구조는 백엔드 LangGraph가 고정.
  UI는 그 구조를 **시각화·설정**할 뿐, 임의 DAG 편집은 비목표 (백엔드 그래프와 불일치 위험).
- ❌ 노드 위치 영구 저장 — 레이아웃은 매번 자동 계산.
- ❌ 엣지 조건 분기 편집 — 압축 루프 등 제어 흐름은 백엔드 소관.

## 결과

### 긍정적
- ✅ JD분석 → 항목별 병렬 파이프라인이 한눈에 보임 (멀티에이전트 차별점 가시화)
- ✅ 각 노드에서 모델 설정 → 생성까지 한 화면 (ADR-025 항목별 설정의 UI 기반)
- ✅ 실시간 상태 체인으로 어느 단계가 도는지/막혔는지 즉시 파악
- ✅ 포트폴리오 데모에서 강력한 시각적 임팩트

### 부정적
- ⚠️ 클라이언트 전용 컴포넌트 → SSR hydration 이슈 주의 (localStorage 지연 로드로 해결)
- ⚠️ SSE 이벤트 ↔ 그래프 상태 수동 매핑 유지보수 비용
- ⚠️ 모바일 미대응
