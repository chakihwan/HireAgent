# ADR-026: 자가 평가 루브릭화 + 생성 과정 투명성

- **상태**: 채택 (Accepted)
- **날짜**: 2026-06-02
- **결정자**: 개발자
- **관련**: [ADR-015](015-langgraph-send-item-subgraph.md), [ADR-022](022-essay-output-defense-layers.md), [ADR-001](001-char-count-validation.md)

---

## 컨텍스트

자소서 생성 파이프라인은 `retrieve → write → [compress ≤3] → evaluate`로 동작한다.
실사용 중 두 가지 신뢰성 문제가 드러났다.

### 문제 1 — 평가 점수의 기준이 없음

기존 `evaluator`는 LLM에게 단순히 **"전문가인 척 1~10점을 매겨라"**고만 지시했다.

```python
_SYSTEM = "당신은 자기소개서 평가 전문가입니다. { "score": <1-10 정수>, ... } JSON만 출력하세요."
```

- **채점 기준 부재**: 무엇을 보고 몇 점인지 코드가 설명 불가. 모델 내부 판단에 전적 의존.
- **점수 쏠림**: 루브릭 없으면 LLM은 7~8점에 몰아주는 관대한 채점 경향.
- **재현성 낮음**: 같은 자소서도 실행마다 점수가 흔들림.
- **파싱 실패 = 7.0 위장**: JSON 파싱 실패 시 `7.0` 고정 → "평가 실패"가 "7점"으로 둔갑.

### 문제 2 — 생성 과정이 블랙박스

압축 루프(ADR-015, 최대 3회)가 돌지만 결과 카드는 **최종본만** 보여줬다.

- "글자수 조정이 실제로 되는지" 사용자가 알 수 없음.
- 500자 목표인데 결과가 600자+로 나와도, 압축이 동작했는지/실패했는지 불투명.
- 디버깅 시 어느 단계에서 글자수가 튀는지 추적 불가.

---

## 결정

**평가는 명시적 루브릭으로, 생성 과정은 단계별 이력으로 투명화한다.**

### 결정 1 — 5항목 루브릭 채점 (`evaluator`)

각 항목 0~2점, 합산 10점:

| 항목 | 기준 |
|------|------|
| 직무적합 (job_fit) | 공고 요구역량·인재상과 경험이 연결되는가 |
| 구체성 (specificity) | 추상적 수식어 대신 구체적 경험·수치가 있는가 |
| 진정성 (authenticity) | 경험 자료 근거, 과장·날조 없는가 (ADR-022 연계) |
| 흐름 (flow) | 도입·전개·마무리가 자연스러운가 |
| 가독성 (readability) | 문장이 명료하고 군더더기 없는가 |

핵심 설계:
- **JD 분석을 평가 프롬프트에 포함** → "직무적합"을 실제 공고와 비교 가능.
- **총점은 LLM이 아니라 백엔드가 합산** → LLM 산수 실수 차단. LLM은 항목 점수만 출력.
- **항목 점수 0~2로 clamp** → 범위 이탈 방어.
- **파싱 실패 시 `None`** (7.0 고정 폐기) → "평가 실패"를 명시. 프론트는 `score !== null`로 처리.
- 항목별 점수를 `evaluation_scores` dict로 끝까지 전달 → 막대그래프 시각화.

### 결정 2 — 단계별 이력 (`draft_history`)

write/compress 각 노드가 `{step, iteration, content, char_count, char_target}`를
`draft_history`에 누적 (`Annotated[list, operator.add]` reducer).

- `DraftResult`에 포함 → 결과 카드 "단계별 이력" 토글로 초안→압축→최종을 순서대로 표시.
- 각 단계에 글자수 진행 바(목표 대비) + 본문 전문.

---

## 이유

### 왜 루브릭인가
- **설명 가능성**: "8점 = 직무적합2+구체성2+진정성1+흐름1+가독성2" 처럼 점수에 근거가 생김.
- **재현성**: 명시 기준이 모델 판단의 분산을 줄임.
- **개선 방향 제공**: 약한 항목(1점)이 곧 사용자가 보강할 지점.

### 왜 총점을 백엔드가 합산하나
- 작은 LLM은 "2+2+1+1+2"를 8이 아닌 다른 값으로 틀리게 적기도 함 (ADR-001의 "LLM은 셈에 약하다" 원칙과 동일 맥락). 항목 점수만 받고 합산은 Python이 한다.

### 왜 이력을 노출하나
- 압축 루프가 LLM 한계로 **수렴 실패**하는 경우가 있는데(글자수 토큰 인식 문제), 이력이 없으면 "압축이 동작 안 한다"는 오해를 부른다. 이력은 **"동작하지만 한계가 있다"**를 보여줘 사용자가 인라인 편집으로 마무리하도록 유도.
- 멀티에이전트 파이프라인의 중간 산출물을 보여주는 것 자체가 제품의 투명성·신뢰 차별점.

---

## 트레이드오프

| 항목 | 결정 |
|------|------|
| 평가 프롬프트 길이 증가 | 루브릭 + JD 분석으로 +토큰 — 평가 품질 위해 수용 |
| 응답 페이로드 증가 | `draft_history`가 본문을 단계 수만큼 중복 — 보통 2~4단계라 허용 |
| 루브릭도 결국 LLM 판단 | 항목별이라도 채점은 LLM. 완전 객관은 아님 — 단 분산은 감소 |
| 글자수 미수렴은 미해결 | 이력으로 가시화만 했고 근본 해결(Python 트런케이션)은 품질 희생이라 보류 |

---

## 명시적으로 하지 않는 것

- ❌ 평가 점수 기반 자동 재작성 루프 (피드백→재생성) — 향후 과제, 현재는 점수·약점만 제공
- ❌ Python 강제 트런케이션으로 글자수 100% 맞추기 — 문장 잘림 품질 희생, 인라인 편집이 워크어라운드
- ❌ 루브릭 가중치 차등 (현재 5항목 균등 2점)

---

## 결과

### 긍정적
- ✅ 점수에 근거가 생겨 "왜 8점인지" 설명 가능 (텍스트 + 막대그래프)
- ✅ 총점 합산 정확 (검증: 2+2+1+1+2=8.0)
- ✅ 압축 과정이 투명 — 동작/한계가 한눈에
- ✅ 파싱 실패가 7점으로 위장되지 않음

### 부정적
- ⚠️ 루브릭 채점도 LLM 의존이라 완전 객관·재현은 아님
- ⚠️ 글자수 미수렴은 여전 (이력으로 보이기만 함) — exaone 모델 한계
