# ADR-032: 직무 적합도 지도 — 요구 ↔ 경험 매칭 시각화

- **상태**: 채택 (2026-06-18 구현)
- **날짜**: 2026-06-18
- **결정자**: 개발자
- **관련**: [ADR-031](031-conversational-stepwise-execution.md)(대화형 단계 실행 — "내 경험" 단계), [ADR-017](017-kure-v1-embedding.md)(KURE-v1 임베딩), 요구사항 §1.4(프로젝트 RAG 차별점)

---

## 컨텍스트

ADR-031의 "내 경험" 단계는 직무 분석 후 RAG로 내 경험(프로젝트 청크)을 검색해 옵시디언풍
뉴런 맵으로 보여줬다. 프로젝트=허브, 청크=위성, **유사도로 거리·크기·활성**.

실사용 피드백:

> "예쁘긴 한데 — 이게 **왜 이 직무에 맞는지**는 안 알려줘. 그냥 '가깝다'만 보여."

맞는 지적이다. 유사도 % 나열은 RAG의 진짜 가치(왜 이 경험을 끌어왔는가)를 숨긴다. 그리고
"내 경험이 이 직무를 **얼마나 커버하는가 / 어디가 비었는가**"라는, 지원자가 정말 궁금한 질문에
답하지 못한다.

---

## 결정

뉴런 맵을 **"경험 지도"에서 "직무 적합도 지도"로** 전환한다.

```
          [멀티에이전트]──── HireAgent
         /                      \
     🎯JD ──[Python 백엔드]──── GeoLogistics
         \                      /
          [Kubernetes]  ← 보강 필요(주황, 매칭 경험 없음)
```

- **중심**: 지원 직무(회사)
- **안쪽 고리**: JD 핵심 요구 역량 (3~5개)
- **바깥 고리**: 내 경험(프로젝트 허브) — 켜고 끄는 큐레이션 단위 유지
- **엣지**: 요구 ↔ 경험을 **매칭 점수**로 연결 (두꺼울수록 강함). 한 경험이 여러 요구에
  걸치면 교차 연결 → 신경망 느낌
- **보강 필요(gap)**: 임계(0.4) 이상 받쳐주는 경험이 없는 요구 = 주황 강조. "이 직무가
  원하지만 내 경험에 약한 영역"을 한눈에

### 요구 추출은 LLM 미사용 (결정론적)

`jd_analyzer`는 이미 `## 핵심 요구 역량`을 불릿 리스트로 출력한다(ADR-031 분석 노드).
→ **정규식으로 불릿만 추출**한다. 추가 LLM 호출 없음(비용·지연 0), 추출 결과 결정론적
(Rule #1 "셀 수 있는 건 Python으로"의 정신과 일치).

### 매칭은 프로젝트 다양성 검색 재사용

요구마다 `search_grouped_by_project(query=요구, per_project=1)`(ADR-031 D 도입)로 프로젝트별
최적 청크를 찾아 점수를 매긴다. 청크 많은 레포가 독식하지 않고 모든 경험이 공정하게 평가됨.

---

## 근거

- **RAG 가치 노출** — "왜 이 경험인가"를 요구 단위로 설명(explainable retrieval). 포트폴리오에서
  멀티에이전트/RAG 역량을 보여주는 가장 강한 서사.
- **행동 유도** — 단순 유사도보다 "MLOps가 비었네 → 그 항목은 보강하자"가 명확. 지원 전략에 직결.
- **저비용 구현** — 요구는 기존 JD 분석 산출물에서 정규식 추출, 매칭은 기존 검색 함수 재사용.
  새 LLM 호출·새 모델 선택 UI 없음.

## 대안

- **(기각) 비주얼만 다듬기**(곡선 엣지·라벨) — 효과 대비 품이 크고, 근본 질문("왜 맞는가")엔
  답 못 함.
- **(기각) LLM으로 요구 추출** — 정규식으로 충분(JD 분석이 이미 구조화 출력). 호출·지연·비용 낭비.
- **(보류) 청크 단위 충족도** — 요구↔청크까지 그리면 과밀. 프로젝트(경험) 단위가 큐레이션·가독성에 맞음.

## 영향

- 백엔드: `POST /nodes/coverage/run`(`_extract_requirements` 정규식 + 요구별 매칭). 스키마
  `CoverageRequest/RequirementCoverage/CoverageMatch/CoverageResponse`.
- 프론트: `ExperienceNeurons`를 적합도 지도로 재작성(요구 노드 + gap 주황 + 매칭 엣지). "내 경험"
  단계에서 `runRagSearch`와 `runCoverage` 병렬 호출, 경험 켜고 끄기·작성 인용은 ADR-031 그대로.
- 요구 추출 실패 시(불릿 없음) 경험만 단순 방사형으로 폴백 → 회귀 없음.
- 후속 다듬기(2026-06-23): "충족도"→**"적합도"** 명칭 통일, **선은 평소 유령처럼 흐리고 hover 시 관련
  요구·경험만 강조**(거미줄 방지), 요구 노드는 적합도 진행바, 경험 라벨 `owner/` 제거, 크게보기 모달
  확대 + 좌드래그 패닝. 깜빡임은 `useNodesState`로 위치 고정 + hover 시 `data.faded`·선 스타일만
  in-place 패치해 해결(노드 재마운트 0).

## 향후

- 요구 노드 클릭 → 그 요구를 충족할 경험만 강조/필터
- gap 요구에 "이 강점으로 대체 서술" 제안 (작성 프롬프트에 반영)
- 충족도 종합 점수(직무 적합도 한 줄 요약)
